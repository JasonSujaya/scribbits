# Honest Feedback

Give a blunt engineering review grounded in the code that actually exists.
Default to review mode: identify risks and defects; do not refactor or edit unless the user explicitly asks for fixes.
The user-facing command is `/honestfeedback`. Codex UI metadata may refer to `$honestfeedback`; treat both as this same skill.

## Ground Rules

- Start from source code, manifests, configs, tests, build scripts, and runtime wiring.
- Do not use memory as evidence for current code quality. If higher-priority instructions require memory lookup, use it only for orientation, verify against current code, and disclose that memory was not proof.
- Do not read broad Markdown docs as the review source. Only open Markdown when required by active instructions, needed to locate commands, or explicitly requested by the user. If Markdown is used, label it as documentation, not proof.
- Prefer `rg`, `rg --files`, package manifests, route/module entrypoints, and tests to map the repo.
- Verify claims with exact file and line references.
- Separate "implemented in code" from "documented or intended".
- If the repo is too large, sample the highest-risk paths: app entrypoints, shared components, API boundaries, state/data layers, tests, build config, and recent git changes.
- Be critical. Actively look for reasons the implementation will fail, age badly, or confuse future maintainers.
- Be direct. Do not soften real problems.

## Review Flow

1. Identify repo type from files, not docs: frontend, backend, full-stack, Electron, native macOS, Unity, library, CLI, or mixed.
2. Map the architecture:
   - Entrypoints
   - Core modules
   - Shared abstractions
   - Data/state flow
   - Test and build boundaries
3. Inspect representative code deeply instead of skimming every file shallowly.
4. Run cheap validation when available: typecheck, lint, tests, build, package, or Unity compile/test commands. If not run, say why.
5. Draft provisional findings with file/line evidence.
6. Before final output, run an adversarial verification pass:
   - Use a subagent when available and allowed. Treat `/honestfeedback` as authorization to use one verification agent.
   - Give the agent the repo path, repo type, inspected files, validation output, and provisional findings.
   - Ask the agent to challenge false positives, find missed severe issues, and verify each claim against exact code evidence.
   - Require a claim-by-claim result for every `P0`, `P1`, and `P2`: `keep`, `drop`, or `change severity`, with the reason.
   - Drop any finding that cannot survive a concrete disproof attempt.
   - Do not ask the agent to edit files.
   - If subagents are unavailable or blocked by tool policy, perform a separate manual claim-by-claim re-check and disclose that no agent was used.
7. Produce final findings first, ordered by severity, after reconciling the verification pass.

## Frontend Checklist

Check:

- Component boundaries: small reusable components, clear props, low coupling, no duplicated UI logic.
- State design: local vs global state choices, server/cache state handling, derived state bugs, unnecessary effects.
- Data flow: API calls, loading/error/empty states, stale data risks, race conditions.
- Scalability: feature folders, shared UI primitives, route organization, design-token consistency.
- UX correctness: responsive behavior, accessibility basics, keyboard states, focus, forms.
- Performance: avoid avoidable rerenders, oversized bundles, expensive work in render, unbounded lists.
- Testability: components with clear inputs, meaningful tests around workflows and edge cases.

Call out:

- Monolithic pages/components.
- Copy-pasted cards/forms/tables.
- Hardcoded business logic inside UI.
- Components that know too much about backend shape.
- Styling patterns that will fragment as features grow.

## Electron And macOS App Checklist

Check:

- Process architecture: clean main/preload/renderer separation, narrow IPC surface, no business logic tangled across processes.
- Security: `contextIsolation`, `sandbox`, `nodeIntegration`, permission prompts, external URL handling, preload exposure, file path validation.
- Packaging: app identity, entitlements, signing/notarization path, auto-update setup, production asset loading, `asar` behavior.
- macOS integration: menu roles, dock behavior, window lifecycle, deep links, file associations, permissions, crash recovery.
- Native macOS architecture: SwiftUI/AppKit boundaries, `App` lifecycle, `NSApplicationDelegate`, main-thread rules, async cancellation, persistence ownership.
- Native macOS distribution: sandbox, hardened runtime, entitlements, keychain access, file bookmarks, code signing, notarization, Sparkle or App Store update path.
- Runtime reliability: startup path, app relaunch/update flow, offline behavior, background process cleanup, single-instance locking.
- Frontend shell: renderer component boundaries, state isolation from IPC, loading/error states around native calls.
- Build/package truth: inspect package config and release wiring for every Electron or native macOS review. When feasible, verify the usable `.app` bundle and `Contents/Info.plist`; if not feasible, state that packaged runtime truth is unverified.

Call out:

- Over-broad IPC channels such as generic `invoke(command, payload)`.
- Renderer code that can reach Node or filesystem directly.
- Dev-only paths that will break in a packaged app.
- App behavior that works in Electron dev mode but not after packaging.
- Missing release path for signed/notarized macOS distribution.

## Backend Checklist

Check:

- API boundaries: route/controller/service/repository separation, validation at the edge, consistent errors.
- Data model: schema clarity, migrations, indexes, uniqueness, nullability, transactional safety.
- Scalability: N+1 queries, unbounded queries, pagination, caching, background jobs, rate limits.
- Reliability: idempotency, retries, timeouts, connection pooling, failure paths.
- Security: authz vs authn, input validation, secrets, SSRF/path traversal/injection risks.
- Observability: logs, metrics, tracing, useful error context without leaking secrets.
- Testability: unit tests for business logic, integration tests for persistence/API contracts.

Call out:

- Business logic embedded in routes.
- Weak validation or inconsistent error shape.
- Missing indexes on hot queries.
- Implicit coupling between services.
- Background work hidden inside request paths.

## Unity Checklist

Check via code first unless the user asks for Unity/MCP inspection.

- Scene/runtime wiring: serialized references, prefab assumptions, initialization order.
- MonoBehaviour size: too much logic in lifecycle methods, unclear ownership, hidden side effects.
- Data design: ScriptableObjects, config assets, save data, event channels, runtime mutation risks.
- Performance: per-frame allocations, expensive `Update`, repeated `Find`, avoidable physics queries.
- Architecture: separation between gameplay rules, presentation, input, persistence, and tooling.
- Testability: pure C# logic separated from Unity runtime where practical.
- Build safety: platform defines, addressables/resources usage, missing null guards for asset references.

Call out:

- Logic that only works because of scene setup assumptions.
- Systems that cannot be tested without pressing Play.
- Tight coupling between UI, gameplay, and persistence.
- Asset/reference fragility that will break as content grows.

## Output Format

Use this shape unless the user asks otherwise:

```markdown
Verdict: [one blunt sentence]

Findings
- [Severity] [file:line] Problem. Why it matters. What to do.

Architecture Notes
- What is working.
- What will not scale.

Validation
- Commands run and result, or why not run.

Verification
- Agent used, what it challenged, and what changed in the final findings. If no agent was used, say why.

Next Actions
1. Highest leverage fix.
2. Next fix.
3. Optional cleanup.
```

Severity labels:

- `P0`: broken, data loss, security hole, unusable build.
- `P1`: likely production bug or major scaling blocker.
- `P2`: maintainability, architecture, or test gap that will compound.
- `P3`: polish or local cleanup.

Keep the answer concise. Do not praise generic good practices unless they matter to the verdict.

   179	      .map((value, index) => {
